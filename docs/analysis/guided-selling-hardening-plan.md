## Executive Summary

The `prepare_guidedselling` flow currently fails on first run and
requires multiple manual fix-up rounds. The root cause is a **dual-path
conflict**: SFDMU inserts OmniProcess/OmniProcessElement records as
data, then the Metadata API deploy creates the *same* records as
metadata — producing duplicates, orphaned elements, stale
`WebComponentKey` references, and version mismatches.

The fix is to give each object type **one authoritative path** and
eliminate the overlap. OmniScript objects belong to the Metadata API.
Bridge records (`OmniProcessAsmtQuestionVer`) belong to SFDMU. The two
should not compete.

## Root Cause Analysis

### Problem 1: Duplicate OmniProcess Records

The metadata deploy matches existing OmniProcess records by
`Type|SubType|Language`, not by record ID or `UniqueName`. When it finds
a deactivated record (from the SFDMU insert), it creates a **new**
version record instead of updating the existing one.

**Result:** Multiple OmniProcess records with the same `UniqueName`. In
soteraNicole: 5 records named `QuantumBit_GuidedSelling_English_2`. In
vibeQbit: 3 records with the same name.

**Schema fact:** `OmniProcess.UniqueName` is `unique=false`. Salesforce
does not enforce uniqueness — duplicate names are permitted.

**Schema fact:** OmniProcess records cannot be deleted via DML. Salesforce
throws "You cannot update or delete an active OmniScript record" even on
inactive records. Duplicates accumulate permanently.

### Problem 2: Orphaned OmniProcessElement Records

SFDMU inserts OmniProcessElement records pointing to OmniProcess ID "A"
(from the SFDMU-created record). The metadata deploy creates a new
OmniProcess ID "B" with its own inline elements. The SFDMU elements
now point to the inactive record "A" — they're orphaned.

### Problem 3: Stale WebComponentKey

`WebComponentKey` is an org-specific reference to a compiled LWC. When
SFDMU imports this value from a source org, it points to a component
that doesn't exist in the target org.

**Key finding:** Discovery Framework OmniScripts with
`discoveryFrameworkUsageType=2` (guided selling forms) have
`WebComponentKey=null` in both source and target orgs — this is
**normal**. Only `discoveryFrameworkUsageType=0` (integration scripts
like `ProductGuidedSelection_Integration`) need a compiled LWC.

LWC compilation is triggered **only** by Metadata API deploy, not by
DML activation.

### Problem 4: Version Number Mismatch

The `.os-meta.xml` file is named `QuantumBit_GuidedSelling_English_1`
but declares:

```xml
<uniqueName>QuantumBit_GuidedSelling_English_2</uniqueName>
<versionNumber>2.0</versionNumber>
```

The SFDMU CSV references `QuantumBit_GuidedSelling_English_1`. After
the metadata deploy creates a record named `_English_2`, SFDMU can't
resolve the FK.

### Problem 5: Activation Script Filters Wrong Version

`activateGuidedSellingOmniScripts.apex` filters on `VersionNumber = 1`,
but the metadata deploy creates records at version 2 (from the
`.os-meta.xml` content). The activation script misses them.

## Object & Schema Reference

### OmniProcess (sObject)

| Field | Type | Required | Unique | Updateable |
|-------|------|----------|--------|------------|
| UniqueName | string | No | **No** | Yes |
| Type | picklist | No | No | Yes |
| SubType | picklist | No | No | Yes |
| Language | picklist | No | No | Yes |
| VersionNumber | double | No | No | Yes |
| IsActive | boolean | Yes | No | Yes |
| WebComponentKey | string | No | No | Yes |
| DesignerCustomizationType | string | No | No | Yes |
| DiscoveryFrameworkUsageType | picklist | No | No | Yes |

- `UniqueName` follows the pattern `{Type}_{SubType}_{Language}_{VersionNumber}`
- Not enforced as unique — multiple records can share the same value
- Records **cannot be deleted** via DML (permanent accumulation risk)

### OmniProcessElement (sObject)

| Field | Type | Required | References |
|-------|------|----------|------------|
| Name | string | Yes | — |
| OmniProcessId | reference | Yes | OmniProcess (master-detail) |
| ParentElementId | reference | No | OmniProcessElement (self-ref) |
| EmbeddedOmniScriptKey | string | No | — |

- `Name` is unique within a single OmniProcess but NOT globally unique
- Cross-OmniProcess collisions observed: `ProductGuidedSelectionIntegration`,
  `Save-Responses`, `Step1` all appear in multiple OmniProcesses

### OmniProcessAsmtQuestionVer (sObject)

| Field | Type | Required | References |
|-------|------|----------|------------|
| AssessmentQuestionVersionId | reference | **Yes** | AssessmentQuestionVersion |
| OmniProcessId | reference | **Yes** | OmniProcess |
| OmniProcessElementId | reference | **Yes** | OmniProcessElement |
| IsPrefill | boolean | No | — |

- Has **three** required FK fields — all must resolve correctly
- Not a metadata type — no Metadata API support; SFDMU or Apex only
- SFDMU externalId `AssessmentQuestionVersion.Name;OmniProcessElement.Name`
  is all-traversal — subject to SFDMU v5 Bug 5 (Upsert may not match)

### OmniScript Metadata Type

The `OmniScript` metadata type (`.os-meta.xml`) includes:
- OmniProcess-level fields (type, subType, language, isActive, etc.)
- OmniProcessElement definitions (inline as `<omniProcessElements>`)
- Does NOT include `OmniProcessAsmtQuestionVer` bridge records

Deployment behavior:
- Matches target org records by `Type|SubType|Language`
- Sets `isActive` per the XML — compilation happens automatically for
  scripts with `isWebCompEnabled=true`
- Inactive OmniScripts (`<isActive>false</isActive>`) may be skipped
  by the deploy engine
- Creates new version records when deployed to an org with existing
  deactivated records of the same Type/SubType/Language

### AssessmentQuestion Metadata Type

The `AssessmentQuestion` metadata type (`.aq-meta.xml`) includes:
- Question fields (developerName, dataType, questionCategory)
- AssessmentQuestionVersion records (inline as `<assessmentQuestionVersion>`)

### AssessmentQuestionSet Metadata Type

The `AssessmentQuestionSet` metadata type (`.aqs-meta.xml`) includes:
- Set fields (developerName, name)
- AssessmentQuestionAssignment records (via `<assessmentQuestionDeveloperNames>`)

**Current gap:** `qb_guided_selling.aqs-meta.xml` lists only 1 of 4
questions (`solution_type_looking_for`). It should list all 4.

## Metadata API Findings

### OmniScript Deploy Behaviors

1. **Version matching:** Metadata API matches by `Type|SubType|Language`,
   not by `UniqueName` or record ID.

2. **LWC compilation:** Triggered automatically during deploy for scripts
   with `<isWebCompEnabled>true</isWebCompEnabled>`. DML activation does
   NOT trigger compilation.

3. **Inactive OmniScripts:** May be excluded from deployment. The
   `guided_selling_English_1` script sets `<isActive>false</isActive>` —
   needs verification that it deploys successfully.

4. **Version behavior (Metadata API / Standard Runtime):** Keeps the same
   version across orgs. Does NOT create new versions like the Build Tool
   / Package Runtime does. However, when deployed to an org with existing
   deactivated records, a new version record may be created.

5. **Element handling:** OmniProcessElement records are deployed inline
   within the `.os-meta.xml`. The deploy creates/updates elements as
   part of the OmniScript definition — not as separate DML operations.

### WebComponentKey Findings

| discoveryFrameworkUsageType | WebComponentKey | LWC Needed? |
|----------------------------|-----------------|-------------|
| 0 (Integration) | SET (org-specific UUID) | Yes |
| 2 (Guided Selling form) | null | No — rendered via Discovery Framework runtime |

This was confirmed identical in both vibeQbit and soteraNicole.
`WebComponentKey` should NOT be included in SFDMU CSV exports — it's
org-specific and stale in any target org.

### Why Multipass SFDMU Is Not the Right Pattern

The multipass pattern (used in billing, rating, procedure-plans) works
because SFDMU is the **sole authority** for the data. Those objects
(BillingPolicy, UsageResource, ProcedurePlanSection) have no parallel
Metadata API type.

OmniScript objects have a dual identity — data records AND metadata
types. A multipass SFDMU plan cannot replicate what the Metadata API
deploy does (LWC compilation, inline element creation, activation
handling). The correct architecture gives each object type one
authoritative path.

|---|------|----------|------------|
| 1 | UniqueName mismatch between `.os-meta.xml` (`_English_2`) and CSV (`_English_1`) | **Critical** | Align `.os-meta.xml` to v1 or update CSV to v2. Both must match. |

## Proposed Architecture

### Principle

Each object type gets **one authoritative path**:

| Object | Authority | Reason |
|--------|-----------|--------|
| OmniProcess | Metadata deploy (`.os-meta.xml`) | Handles activation + LWC compilation |
| OmniProcessElement | Metadata deploy (inline in `.os-meta.xml`) | Created as part of OmniScript definition |
| AssessmentQuestion | Metadata deploy (`.aq-meta.xml`) | Proper metadata type |
| AssessmentQuestionVersion | Metadata deploy (inline in `.aq-meta.xml`) | Created as part of question definition |
| AssessmentQuestionSet | Metadata deploy (`.aqs-meta.xml`) | Proper metadata type |
| AssessmentQuestionAssignment | Metadata deploy (inline in `.aqs-meta.xml`) | Created via `assessmentQuestionDeveloperNames` |
| OmniProcessAsmtQuestionVer | **SFDMU** | No metadata API support — data-only sObject |
| AssessmentQuestionConfig | Neither (auto-generated) | Side effect of metadata deploy |
| AssessmentQuestionSetConfig | Neither (auto-generated) | Side effect of metadata deploy |
| OmniScriptConfig | Neither (auto-generated) | Side effect of metadata deploy |

### Revised Flow

```
prepare_guidedselling:
  1: deploy_post_guidedselling
     when: project_config.project__custom__guidedselling
     # Deploys: OmniScripts (with elements + activation + LWC compilation),
     # AssessmentQuestions (with versions), AssessmentQuestionSets (with
     # assignments), ProductDiscovery settings.

  2: insert_qb_guidedselling_data
     when: project_config.project__custom__guidedselling and project_config.project__custom__qb
     # SFDMU: Inserts OmniProcessAsmtQuestionVer bridge records only.
     # Readonly configs for validation.
```

**Removed steps:**
- `deactivate_guidedselling_omniscripts` — not needed; metadata deploy
  handles activation state
- `activate_guidedselling_omniscripts` — not needed; metadata deploy
  sets `<isActive>true</isActive>` and triggers LWC compilation

### Revised `export.json`

```json
{
  "apiVersion": "66.0",
  "excludeIdsFromCSVFiles": "true",
  "objectSets": [
    {
      "name": "Guided Selling Bridge Data",
      "objects": [
        {
          "query": "SELECT DeveloperName, Language, MasterLabel FROM AssessmentQuestionConfig",
          "operation": "Readonly",
          "externalId": "DeveloperName"
        },
        {
          "query": "SELECT DeveloperName, Language, MasterLabel FROM AssessmentQuestionSetConfig",
          "operation": "Readonly",
          "externalId": "DeveloperName"
        },
        {
          "query": "SELECT DeveloperName, Language, MasterLabel FROM OmniScriptConfig",
          "operation": "Readonly",
          "externalId": "DeveloperName"
        },
        {
          "query": "SELECT AssessmentQuestionVersionId, AssessmentQuestionVersion.Name, IsPrefill, OmniProcessElementId, OmniProcessElement.Name, OmniProcessId, OmniProcess.UniqueName FROM OmniProcessAsmtQuestionVer",
          "operation": "Insert",
          "externalId": "AssessmentQuestionVersion.Name;OmniProcessElement.Name",
          "deleteOldData": true
        }
      ]
    }
  ],
  "orgs": []
}
```

> **Note:** `OmniProcessAsmtQuestionVer` uses `Insert + deleteOldData: true`
> instead of `Upsert` because the externalId is composed entirely of
> relationship traversals (SFDMU v5 Bug 5 — Upsert cannot match, inserts
> duplicates on re-run). `deleteOldData: true` ensures idempotency by
> deleting all existing records before inserting. This is a leaf table
> with only 4 records and no downstream FK dependencies.

## Appendix: Comparison with Other Multipass Plans

The multipass SFDMU pattern (used in billing, rating, procedure-plans)
was evaluated and rejected for guided selling because:

1. All multipass plans in this repo work because SFDMU is the **sole
   authority** — no competing Metadata API type exists for BillingPolicy,
   UsageResource, or ProcedurePlanSection.
2. OmniProcess has a dual identity (data record AND metadata type).
   SFDMU cannot replicate LWC compilation, which only the Metadata API
   deploy triggers.
3. A multipass SFDMU plan would still require a post-load metadata deploy
   for compilation — reintroducing the dual-path conflict.
