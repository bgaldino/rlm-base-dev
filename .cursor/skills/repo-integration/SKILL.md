# Repository Integration Guide

Use this skill when adding new features, code, metadata, data plans, or
capabilities to the rlm-base-dev repository.

## Quick Rules

1. `force-app/` = step 5. `unpackaged/post_*/` = feature steps. `templates/` = step 29.
2. Never edit `unpackaged/post_ux/` — edit `templates/` instead.
3. Never add `layoutAssignment` to `force-app/` profiles — use `templates/profiles/`.
4. CCI: `--org beta`. SF CLI: `--target-org rlm-base__beta`. Never mix them.
5. Metadata deploys before data loads. Products (PCM) before everything else.
6. Delete rates before rating (FK constraint). Deletion = reverse dependency order.
7. After `cumulusci.yml` edits: `python scripts/ai/generate_cci_reference.py`

## DO NOT

- **DO NOT** edit `unpackaged/post_ux/` — fully regenerated by `assemble_and_deploy_ux`
- **DO NOT** add `layoutAssignment` or `applicationVisibilities` to `force-app/` profiles
- **DO NOT** add `EmailTemplatePage` flexipages to `templates/flexipages/`
- **DO NOT** add object `.object-meta.xml` with `actionOverrides` back to `force-app/`
- **DO NOT** change `Upsert` to `Insert` + `deleteOldData: true` without user approval

## Where Does My Code Go?

| I need to add... | Where it goes |
|-----------------|---------------|
| Foundational metadata (Apex, LWC, fields) | `force-app/main/default/<type>/` |
| Feature-specific metadata | `unpackaged/post_<feature>/` |
| Pre-deploy fields/settings/PSGs | `unpackaged/pre/<numbered_subdir>/` |
| Flexipage (new override) | `templates/flexipages/standalone/<feature>/` |
| Flexipage (additive patch) | `templates/flexipages/patches/<feature>/` |
| Layout (base) | `templates/layouts/base/` |
| Layout (feature) | `templates/layouts/<feature>/` |
| Compact layout / list view | `templates/objects/<feature>/` |
| App definition / actionOverride patch | `templates/applications/` or `patches/<feature>/` |
| Profile (full) | `templates/profiles/base/` |
| Profile (feature patch) | `templates/profiles/patches/<feature>/` |
| Data plan (SFDMU) | `datasets/sfdmu/qb/en-US/qb-<plan>/` |
| Apex script | `scripts/apex/<name>.apex` |
| Python CCI task | `tasks/rlm_<name>.py` |
| Robot E2E test | `robot/rlm-base/tests/e2e/<name>.robot` |
| Robot setup automation | `robot/rlm-base/tests/setup/<name>.robot` |
| Context definition plan | `datasets/context_plans/<Name>/manifest.json` |
| Constraint model data | `datasets/constraints/qb/<Name>/` |

## Drift Capture and Writeback

When UX changes are made directly in the org, use the drift capture workflow to
update templates:

```bash
# 1. Capture drift (retrieve org state + diff against templates)
cci flow run capture_ux_drift --org dev-sb0

# 2. Review drift_report.json, then apply to templates
cci flow run apply_ux_drift --org dev-sb0
```

The writeback task computes `new_base = org_state - patches` and also
auto-updates YAML patch files. Profile writeback requires manual oversight.

For full details, see `docs/features/dynamic-ux-assembly.md`.

## How Do I Configure Something?

| I need to... | Mechanism | Example |
|-------------|-----------|---------|
| Deploy declarative metadata | Metadata API deploy | `force-app/` or `unpackaged/post_*/` |
| Set an org setting (Settings XML) | Settings metadata | `unpackaged/pre/2_settings/` |
| Toggle a Setup page control (no API) | Robot Framework setup test | `enable_document_builder.robot` |
| Load bulk records | SFDMU data plan | `datasets/sfdmu/qb/en-US/qb-pricing/` |
| Create context definition attributes | Connect API (Python task) | `tasks/rlm_context_service.py` |
| Create email templates (no Metadata API) | REST API sidecar task | `tasks/rlm_create_approval_email_templates.py` |
| Activate/deactivate records | Apex script or management task | `scripts/apex/activateRatingRecords.apex` |
| Set App Launcher order | Aura XHR (Robot task) | `reorder_app_launcher.robot` |
| Patch org-specific values for deploy | Patch-then-deploy pattern | `patch_network_email_for_deploy` |

## Adding a New Feature — Checklist

For the complete step-by-step guide with code templates, YAML examples,
and dependency ordering details, read
`.cursor/skills/repo-integration/new-feature-guide.md`.

Summary checklist:
1. Add feature flag in `cumulusci.yml` under `project.custom`
2. Place metadata in correct directory (see table above)
3. Create data plan if needed (export.json + CSVs)
4. Register CCI tasks (insert, delete, extract, idempotency)
5. Add permission sets if needed (PS anchor + assign task)
6. Create context definition plan if needed
7. Create `prepare_<feature>` sub-flow
8. Wire into `prepare_rlm_org` (respect dependency ordering)
9. Add UX changes to `templates/` if needed
10. Regenerate CCI references
11. Update `.forceignore` if UX files moved to `templates/`

## Dependency Ordering

For detailed ordering rules (metadata deploy ordering, profile/object
strip-and-build, data dependencies, activation patterns), read
`.cursor/skills/repo-integration/dependency-ordering.md`.

Key rules:
- `unpackaged/pre/` numbered subdirs deploy in order (1→2→3→5)
- `force-app/` deploys as one bundle at step 5
- `force-app/` profiles are classAccesses-only (full profile at step 29)
- Products (PCM) → pricing → billing → tax → rating → rates
- Delete in reverse: rates → rating → billing → pricing → PCM
